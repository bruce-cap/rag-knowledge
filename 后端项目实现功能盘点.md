# RAG 后端项目实现功能盘点

## 1. 项目概览

当前后端项目是一个基于 Spring Boot 3 的私有知识库/RAG 服务端，已经完成了以下核心能力：

- 用户注册、登录、退出与 JWT 鉴权
- 基于知识空间的权限隔离
- 公共知识空间初始化与自动入会
- 知识空间、成员、文件夹的管理
- 文档上传、异步解析、向量化入库、向量检索
- 基于会话的聊天与流式输出
- RAG 模式下按知识空间权限进行检索增强问答
- 聊天会话与消息管理
- 空间加入申请与审批
- 基础异常处理与单元测试覆盖

从代码实现来看，这个项目已经不是“空壳脚手架”，而是具备完整主链路的后端系统，尤其是“知识空间 + 文档上传 + 向量检索 + RAG 问答 + 权限控制”这一条业务链已经打通。

## 3. 已实现的核心业务模块

## 3.1 用户认证与鉴权

对应代码：

- `src/main/java/com/example/ragbackend/controller/AuthController.java`
- `src/main/java/com/example/ragbackend/service/impl/UserServiceImpl.java`
- `src/main/java/com/example/ragbackend/config/SecurityConfig.java`
- `src/main/java/com/example/ragbackend/config/JwtAuthenticationFilter.java`
- `src/main/java/com/example/ragbackend/utils/JwtUtils.java`
- `src/main/java/com/example/ragbackend/utils/SecurityUtils.java`

已实现功能：

- 用户注册
  - 校验用户名、邮箱、密码
  - 密码要求为 8-16 位字母数字组合
  - 邮箱格式校验
  - 用户名/邮箱唯一性校验
  - 使用 BCrypt 加密密码
- 用户登录
  - 按用户名查询用户
  - 校验密码
  - 生成 JWT
  - 同时把 JWT 放入返回体和 `jwt_token` Cookie
- 用户退出
  - 通过清空 Cookie 实现
- 全局接口鉴权
  - `/api/auth/**` 免鉴权
  - 其余接口默认要求认证
- JWT 解析方式
  - 优先从 `Authorization: Bearer ...` 读取
  - 其次从 Cookie 读取
- 安全上下文能力
  - 可获取当前登录用户 ID
  - 可判断是否为管理员/超级管理员

实现结论：

- “认证体系”已经落地可用
- 使用 JWT 无状态认证
- 前后端可以同时支持 Header Token 与 Cookie Token 两种传递方式

需要注意：

- `SecurityUtils.isAdmin()` 当前实际复用了 `isSuperAdmin()` 逻辑
- 也就是说，全局“管理员”判定本质上等价于角色为 `ADMIN` 或 `SUPER_ADMIN`
- 普通用户在空间内的 `ADMIN` 身份，是空间成员角色，不等于全局后台管理员

## 3.3 知识空间管理

对应代码：

- `src/main/java/com/example/ragbackend/service/impl/KnowledgeSpaceServiceImpl.java`

已实现功能：

- 查询我可访问的知识空间
  - 普通用户：返回其加入过的空间
  - 超级管理员：返回全部空间
- 查询全部可申请加入的空间
  - 普通用户：返回启用中、非系统空间、且自己尚未加入的空间
  - 超级管理员：返回全部启用中、非系统空间
- 创建知识空间
  - 仅超级管理员可创建
  - 创建者自动成为该空间管理员
- 更新知识空间
  - 可修改名称、描述、状态
  - 仅超级管理员可操作
- 删除知识空间
  - 仅超级管理员可删除

  - 仅超级管理员可把成员调整为 `ADMIN` / `MEMBER`

实现结论：

- 空间管理能力已经完整
- 角色规则、删除保护、系统空间保护都已实现

## 3.6 文档上传、存储、解析与删除

对应代码：

- `src/main/java/com/example/ragbackend/controller/DocumentController.java`
- `src/main/java/com/example/ragbackend/service/impl/DocumentServiceImpl.java`
- `src/main/java/com/example/ragbackend/service/impl/DocumentProcessingServiceImpl.java`
- `src/main/java/com/example/ragbackend/config/MinioConfig.java`
- `src/main/java/com/example/ragbackend/config/AsyncConfig.java`

已实现功能：

- 上传文档
  - 接收 Multipart 文件
  - 要求指定知识空间
  - 可选指定文件夹
  - 校验上传目标空间与文件夹归属
  - 将原始文件上传到 MinIO
  - 在数据库写入文档记录
  - 文档初始状态设为待处理
- 异步文档处理
  - 事务提交后异步触发
  - 从 MinIO 拉取原始文件
  - 使用 Apache Tika 解析文档文本
  - 按 `recursive(500, 50)` 规则切分文本片段
  - 为每个片段补充 metadata
    - `document_id`
    - `user_id`
    - `space_id`
    - `folder_id`
  - 使用 DashScope 生成 embedding
  - 分批写入 Chroma
  - 更新文档状态为处理中/成功/失败
  - 失败时记录 `error_message`
- 查询文档列表
  - 支持按空间、文件夹筛选
  - 普通用户只可看到自己有权限访问的空间中的文档
- 删除文档
  - 文档上传者或管理员可删除
  - 删除 MinIO 文件
  - 删除 Chroma 中对应向量
  - 删除数据库记录

文档状态含义：

- `0`：待处理
- `1`：处理中
- `2`：处理完成
- `3`：处理失败

实现结论：

- 文档从“上传原文件”到“向量化入库”完整打通
- 已具备真正的知识入库能力，不只是元数据管理

当前边界：

- 删除文档时实际是物理删除数据库记录，不是软删除
  - 虽然表结构里有 `is_deleted`、`delete_time` 字段
  - 但业务实现使用的是 `removeById`
- 未看到文档下载、预览、重新处理、批量导入等能力

## 6. 当前项目的功能成熟度判断

如果从“实现状态”来判断，当前后端已经完成的不是单点功能，而是一个较完整的 MVP 后端。

### 6.1 已经打通的主链路

主链路 1：用户与权限

- 注册
- 登录

## 7. 当前代码中能看到的限制与风险

以下内容不是“未实现”，而是“已实现基础能力，但仍有明显边界或后续建议”的部分。

### 7.1 会话删除可能遗留消息数据

- `ChatSessionServiceImpl.deleteSession()` 只删除会话表记录
- 没有显式删除对应消息
- 如果数据库没有配置级联删除，可能产生孤儿消息

### 7.2 文档软删除字段未真正使用

- `document` 表有 `is_deleted`、`delete_time`
- 实际删除逻辑是直接 `removeById`
- 当前更像物理删除方案

### 7.3 文件夹父子关系缺少更深层环检测

- 只校验了不能把自己设为自己的父节点
- 未看到“不能挂到自己子孙节点下”的完整校验

### 7.4 权限命名上存在“全局管理员”和“空间管理员”的语义混用

- `SecurityUtils.isAdmin()` 实际等于全局超级管理员判定
- 而空间管理员是 `space_member.role = ADMIN`
- 两者概念不同，后续建议在命名层面进一步区分

### 7.5 缺少用户管理后台能力

- 当前只有认证接口
- 没有看到用户列表、用户详情、用户角色调整、用户禁用等后台管理接口

### 7.6 缺少文档运维类能力

- 未看到重新解析
- 未看到批量删除/批量上传
- 未看到文档预览/下载
- 未看到解析进度查询明细

### 7.7 测试覆盖仍偏服务层

- 已有单元测试是加分项
- 但控制器联调、文档处理异步链路、MinIO/Chroma 集成测试还比较少
